# 运营后台举报证据快照说明

日期：2026-06-30

## 1. 目标

举报处理不能只依赖“当前还能查到的动态、评论或地点点评”。一旦运营隐藏、删除内容，或者后续接入申诉流程，后台必须能追溯当时被举报的内容状态。

本能力把举报证据从动态计算改为固化保存：

- 用户提交举报时，立即保存一份 `evidenceSnapshot`。
- 运营处理有效举报时，如果历史举报没有快照，会先补齐快照。
- 处罚建议和处罚记录复用同一份快照，避免举报中心、内容安全池和处罚中心口径不一致。

## 2. 快照字段

当前快照保存在 `socialReports[].evidenceSnapshot`，字段包括：

- `snapshotId`：快照 ID。
- `snapshotAt`：快照固化时间。
- `reportId`：举报 ID。
- `reporterPhone` / `reporterName`：举报人。
- `ownerPhone` / `ownerName`：被举报内容作者。
- `targetType`：被举报对象类型，例如 `post`、`comment`、`place_review`。
- `targetId`：被举报对象 ID。
- `targetLabel`：后台展示用对象名称。
- `targetStatus`：快照时目标状态。
- `contentText`：被举报内容文本摘要。
- `mediaUrls`：最多 6 张相关图片 URL。
- `reportContent`：举报补充说明。
- `createdAt`：举报创建时间。

## 3. 后台展示

举报中心的“内容快照”列展示：

- 快照对象名称。
- 是否已固化，以及固化时间。
- 快照时目标状态。
- 图片数量。
- 内容摘要。

旧举报如果没有 `snapshotId` 和 `snapshotAt`，后台会提示为动态预览；处理该举报为有效、有效并隐藏或有效并删除时会自动固化。

## 4. 导出

举报 CSV 新增字段：

- 证据快照 ID。
- 证据快照时间。
- 快照目标状态。
- 证据内容摘要。

导出仍走 `/admin/exports/reports.csv`，并写入 `data.export.download` 审计。

## 5. 移动端影响

没有直接移动端展示变化。

间接影响：

- 有效举报仍会通知举报人和被举报作者；地点点评举报被隐藏或删除后，App 地点详情公开点评列表不再展示该点评。
- 按举报建议创建处罚时，处罚记录会保留同一份证据快照。
- 后续用户申诉时，后台可以用这份快照复核处罚依据。

## 6. 当前边界

- 当前只保存图片 URL，不保存图片二进制。
- 当前不保存完整截图渲染结果。
- 当前未接第三方图片审核模型，因此没有图片风险标签。
- 当前未做独立证据快照表，测试后端仍保存在 `socialReports` 内。

生产期建议迁移为独立证据表或对象存储索引：

- `moderation_evidence_snapshots`
- 字段保留 `reportId`、`targetType`、`targetId`、`ownerPhone`、`reporterPhone`、`snapshotAt`、`textHash`、`mediaRefs`、`createdBy`
- 图片原图或缩略图进入对象存储，后台只保存引用和审核标签。

## 7. 待澄清

1. 证据快照生产期需要保留多久，是否至少 1 年？
2. 图片证据是否需要保存平台侧缩略图，而不是只保存 URL？
3. 用户申诉时，是否允许客服把部分证据摘要展示给用户？
4. 删除内容后，作者端是否需要展示“该内容因举报被处理”的占位说明？
5. 第三方内容安全模型回传的图片/文本标签是否也进入同一份快照？
