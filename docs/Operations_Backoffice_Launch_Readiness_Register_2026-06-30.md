# 运营后台上线台账与待澄清清单

日期：2026-06-30

## 1. 目的

这份文档承接运营后台 PRD 里的“上线前必须确认的问题”，并和后台新增的 `上线台账` 页面保持同一套口径。

它解决三个问题：

- 不把测试阶段可用误说成生产级完工。
- 把需要业务、合规或运营拍板的问题集中沉淀。
- 明确后台能力和移动端联动关系，避免后台成为割裂系统。

## 2. 后台入口

新增后台一级菜单：

- `上线台账`

新增接口：

- `GET /admin/launch/readiness`
- `POST /admin/launch/readiness/questions/{questionId}`
- `POST /admin/launch/readiness/signoff`

接口返回：

- `summary`：上线口径、P0 待处理数、待澄清问题数、配置联动数。
- `signoff`：当前上线结论签署、签署快照和快照是否已变化。
- `modules`：后台模块成熟度台账。
- `questions`：上线前必须确认的问题。
- `gaps`：生产风险和预留治理项。
- `linkage`：配置中心联动体检里的关注项。

写接口支持：

- 更新状态：`open`、`reviewing`、`ready`、`deferred`、`closed`。
- 记录负责人/决策来源。
- 记录决策备注。
- 重置人工覆盖记录，回到动态台账默认口径。
- 写入 `adminAuditLogs`，action 为 `launch.readiness.question.update` 或 `launch.readiness.question.reset`。
- 签署上线结论：记录版本、结论、说明、签署人、签署时间和当前台账快照；存在未关闭 P0 时不允许签署 `ready_for_production`。
- 重置上线结论签署，并写入 `launch.readiness.signoff.reset` 审计。

## 3. 页面口径

状态不使用“已完成/未完成”这种粗粒度口径，而是拆成：

- `测试可用`：当前后台和移动端联动已经能支撑真机测试或小范围验证。
- `部分可用`：核心链路已可用，但生产治理能力不完整。
- `生产阻断`：正式上线前必须解决或明确兜底策略。
- `已预留`：后台入口、数据结构或配置项已保留，但不应被误认为上线。

## 4. 当前模块成熟度

第一版上线台账覆盖以下模块：

- 工作台与数据看板。
- 用户排查。
- 宠物档案与宠物日历。
- AI 灵伴生成。
- AI 对话抽检。
- 内容安全任务池。
- 举报与申诉。
- 地点审核。
- 反馈与工单。
- 通知运营。
- 配置中心。
- 数据导出与审计。
- 系统健康与账号权限。

每个模块都记录：

- 后台证据。
- 移动端联动。
- 下一步需要补齐的生产能力。

## 5. 当前 P0 生产阻断

当前仍需重点关注：

- 后台安全：逐账号登录失败锁定、后端 IP 白名单、多管理员账号、第一版角色权限拦截和 TOTP MFA 基座已接入；生产仍需配置真实白名单、为所有活跃后台账号启用 MFA，并补账号禁用/轮换流程。
- 数据底座：当前仍是 JSON state 文件态，但已接入原子写入、滚动 gzip 备份、启动损坏恢复和系统健康检查；正式大规模生产仍建议迁移数据库。
- 内容安全：上线台账会按服务器密钥、文本机审开关、图片机审开关动态计算 P0；三者就绪时，内容安全供应商和图片审核不再标为生产阻断。
- 图片审核：头像、宠物图、宠友圈图片、封面、地点点评图片、新增地点图片、工单附件已进入统一图片审核池；Review 会进入 `pending_review`，Block 会拒绝或驳回公开内容。
- 地点公开点评展示策略：排序、只看有图、后端返回上限和移动端首屏展示条数已接入配置中心并联动 App。
- 高风险配置：配置草稿和风险摘要已接入；维护模式、强制更新、全功能关闭等是否需要双人审批仍待确认。
- 合规文本：App 备案、隐私政策、内容审核制度需要生产版确认。

## 6. 待业务持续澄清

后台页面内置以下待澄清问题：

1. 后台正式域名使用 `ops.lumiiapp.cn`、`admin.lumiiapp.cn`，还是先沿用 `/admin`？
2. 生产后台是否必须白名单 IP？后端能力已接入，配置 `LUMII_ADMIN_IP_ALLOWLIST` 后 `/admin` 页面和 `/admin/*` API 会拦截非白名单 IP；仍需确认生产允许的出口 IP，并建议在网关层同步配置。
3. 后台账号是否接企业微信、飞书或邮箱 MFA？当前已接 TOTP MFA 基座；上线台账会按所有活跃账号是否启用 MFA 自动标记为已接入、部分启用或待配置。
4. 内容安全供应商已选腾讯云；上线台账仅在密钥和文本/图片机审开关未就绪时继续标为待处理。
5. 图片审核策略已落地：Review 先进入待审核/阻止公开发布，Block 拒绝或驳回；后续可继续补用户申诉入口。
6. 私信查看策略已落地：不开放任意全文检索，仅在举报/关系排查中查看最近上下文窗口；窗口、原因必填和保留标记由 `social.messageAccess` 配置，单 admin 阶段为带审计自审批，生产多管理员后可升级双人审批。
7. 用户业务数据清理已接入审批流和高风险最少会签人数；生产期是否开放、隐藏入口或仅保留测试环境仍待确认。
8. AI 失败额度返还规则已接入：默认供应商提交失败、供应商超时和供应商返回失败自动返还；照片不合格、内容安全拦截和运营手动标失败不自动返还，后台人工返还会防重复。
9. 永久封禁是否必须双人审批？高风险最少会签人数已接入，生产值建议配置为 2 或以上。
10. 导出完整手机号是否允许？如允许，谁审批？
11. 地点贡献分已支持用户本人公开徽章、轻量等级、后台手动调整和撤销；是否接排行榜、活动奖励或兑换规则仍待确认。
12. 系统通知是否需要发送审批？频控、通知人群包、对象深链、发送审批和高风险最少会签人数已接入。
13. 配置强制更新、维护模式、全功能关闭是否必须双人审批？当前已支持先保存草稿、提交审批并按高风险最少会签人数发布。
14. App 备案、隐私政策、内容审核制度是否已准备生产版文本？

页面内可为每条问题记录人工决策。人工记录只覆盖问题状态、负责人和备注，不会删除动态证据；重置后恢复系统根据当前配置、健康状态和账号权限计算出的默认口径。

## 7. 移动端联动说明

上线台账不会单独维护一份“看起来很完整”的假数据。

它会读取：

- 配置中心联动体检：`adminConfigLinkageItems()`。
- 系统健康：`adminSystemHealth()`。
- 账号权限：`adminAccounts()`。
- 当前后端业务模块的实现证据。

因此后台页面能看到：

- 哪些配置真实影响 `/app/config`。
- 哪些后端接口会按配置强制拦截。
- 哪些能力目前只是预留或后端单侧生效。

## 8. 上线结论签署

签署不是替代真实问题修复，而是把当前后台判断和运营结论固定成一份可追溯快照。每次签署会记录：

- 发布版本。
- 结论：`not_ready`、`conditional`、`ready_for_test`、`ready_for_production`。
- 签署说明。
- 签署人和签署时间。
- 当前 P0 数、待确认问题数、模块 ready/partial 数和整体上线状态。

如签署后台账里的 P0、待确认问题或整体状态发生变化，页面会标记快照已变化，需要重新签署。

## 9. 暂不做的事情

本版本不做：

- 多管理员分配 owner。
- 高风险最少会签人数、审批通知和值守 SOP。
- 自动生成上线结论。

原因：

- 多管理员账号、基础 RBAC 和高风险最少会签人数已可用，但生产期仍需要明确业务责任、审批通知和值守交接规则。
- 很多问题需要业务策略确认，不能由代码擅自替业务拍板。
- 先把台账、人工决策记录、角色权限和草稿发布治理做成后台可见，后续再接审批 owner、站外通知和值守 SOP。

## 10. 后续建议

下一步可继续沿着这张台账推进：

1. 把 P0 问题逐项关闭。
2. 对每个关闭项补审计证据。
3. 为高风险动作开启审批人分离和最少 2 人会签，并补审批值守通知。
4. 持续复盘腾讯云内容安全样本，补密钥轮换、异步回调、误杀申诉归因和多审核员质检。
5. 为所有活跃后台账号配置 MFA、密码轮换时间和生产网关层 IP 白名单，并补离职交接 SOP。
6. 把 JSON state 迁移到数据库。
