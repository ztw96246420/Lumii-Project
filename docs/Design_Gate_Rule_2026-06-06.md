# Lumii 设计门禁规则

维护日期：2026-06-11

## 当前结论

这份规则继续有效：新增正式页面、新弹层、新复杂状态、新卡片样式和关键交互状态，必须先由用户在 Figma Make 补齐设计；Codex 只负责基于设计源转译实现，不自行发挥视觉风格。

当前缺失页面和提示词统一维护在：

- [Figma_Make_Missing_Page_Prompts_2026-06-06.md](./Figma_Make_Missing_Page_Prompts_2026-06-06.md)

## 核心原则

1. Codex 先输出缺失页面/状态清单和 Figma Make 提示词。
2. 用户在 Figma Make 补齐设计。
3. Codex 基于 Figma Make、源码包或截图对照转译。
4. Codex 只做业务逻辑、状态机、接口和设计还原，不临时创造正式视觉方案。
5. 如必须临时补逻辑，只能作为不可验收视觉的开发 scaffold，不能进入用户验收主链路。

## 允许先做的内容

- 数据类型、mock API、真实 API 契约、状态机。
- 已有页面内的非视觉逻辑修复。
- 已有组件的 bug 修复。
- 不新增页面的路由保护、表单校验、倒计时、toast、loading、缓存、权限状态。
- Android 手势返回、会话恢复、数据同步、后端清理、开发文档更新。

## 必须先等设计的内容

- 新独立页面。
- 新弹窗、确认框、Bottom Sheet。
- 新卡片样式、列表样式、复杂表单布局。
- 聊天富消息样式。
- 地图地点审核、分享、导航确认等新增状态。
- AI 形象反馈、重新生成、多候选、主体选择等新增体验页。
- 安全中心细分流程，例如举报、拉黑、注销账号冷静期。
