# AI 宠物形象：视觉准入与 Flux 分工决策

## 结论

Flux 可以继续用于“真实照片 -> 真实卡通化宠物形象”的生成，也可以在提示词里加入单宠、无人物、保留真实毛色与五官等约束。

但 Flux 不适合作为“人/多宠/无宠物”的唯一判断来源。准入判断需要独立视觉识别模型输出结构化结果，Flux 只负责在通过准入后生成图片。

## 为什么不能只靠 Flux 提示词

1. Flux 的主要目标是生成/编辑图片，不是稳定输出审核结论。
2. 即使提示词要求“有人或多宠不要生成”，接口通常仍会尝试生成一张图，难以作为业务拦截依据。
3. 生成模型可能把人物、多只宠物、其他动物或道具特征混入结果，导致形象不像用户的真实宠物。
4. 我们需要可记录、可复查、可调参的结构化字段，例如 `petCount`、`humanPresent`、`otherAnimalPresent`、`qualityScore`。
5. 先用 Flux 做判断会多烧生成成本，而且失败后用户等待更久。

## MVP 分层

上传照片后：

1. 基础校验：文件、尺寸、格式、是否有图片内容。
2. 视觉准入：后续接真实视觉模型，输出结构化 JSON。
3. 业务分流：单只猫/狗且主体清晰才进入生成；人宠合照、多宠、无宠物默认阻止。
4. Flux 生成：只在准入通过后调用，并在 prompt 中再次强调单宠与真实特征保留。

## 当前实现状态

当前代码已经保留 `analysis` 结构与后端二次校验入口，先支持 mock/规则态流转。后续接视觉模型时，只替换上传分析实现，不需要重写前端流程。
