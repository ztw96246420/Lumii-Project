# 运营后台用户备注与风险标签说明

日期：2026-06-30

## 1. 目标

补齐用户管理中“运营内部上下文”的能力，让客服和运营在用户列表就能看到：

- 这个用户是否被人工标记为测试账号、重点用户、需要回访或违规观察。
- 这个用户最近是否有运营备注。
- 备注和标签是谁改的、为什么改、什么时候改。

该能力只服务后台运营，不面向移动端用户展示。

## 2. 后台入口

页面：用户管理。

列表新增：

- “运营标记”列：展示风险标签、备注数量、最近备注摘要。
- “备注”按钮：追加一条运营备注。
- “标签”按钮：更新该用户的风险标签。

接口：

- `POST /admin/users/{phone}/notes`
- `POST /admin/users/{phone}/risk-tags`

## 3. 风险标签白名单

当前支持以下标签：

| key | 展示名 | 用途 |
| --- | --- | --- |
| `test_account` | 测试账号 | 内测或反复清数据的账号 |
| `key_user` | 重点用户 | 需要重点关注的真实用户 |
| `needs_followup` | 需要回访 | 客服或运营需要主动跟进 |
| `complaint` | 投诉处理中 | 用户投诉还未完全闭环 |
| `watch` | 违规观察 | 内容或行为需要观察 |
| `suspected_harassment` | 疑似骚扰 | 社交骚扰风险 |
| `suspected_abuse` | 疑似违规 | 广义违规风险 |
| `ai_sample` | AI 异常样本 | AI 生成或对话异常样本用户 |

前端输入支持 key 或中文展示名，用逗号、空格、中文顿号分隔。留空表示清空标签。

## 4. 备注规则

- 每条备注最多 500 字。
- 每个用户最多保留最近 30 条备注。
- 备注按创建时间倒序展示。
- 备注只允许追加，不做物理删除入口。
- 用户列表只展示最近一条备注摘要，用户详情接口返回完整 `adminNotes` 列表。

## 5. 数据保存

当前测试后端直接保存在用户账号对象上：

- `users[phone].adminNotes`
- `users[phone].adminRiskTags`

原因：这两类信息属于账号壳和运营内部元数据，不属于用户宠物、AI、社交、地点、通知等可清理业务数据。

生产期建议迁移为独立表：

- `user_ops_notes`
- `user_ops_tags`

这样便于权限、审计、查询和长期保留。

## 6. 审计

添加备注写入：

- `action`: `user.note.create`
- `targetType`: `user`
- `targetId`: 目标手机号
- `before`: 添加前用户摘要
- `after`: 添加后用户摘要与新增备注
- `reason`: 备注内容摘要

更新标签写入：

- `action`: `user.risk_tags.update`
- `targetType`: `user`
- `targetId`: 目标手机号
- `before`: 更新前用户摘要
- `after`: 更新后用户摘要
- `reason`: 更新用户风险标签

## 7. 导出

用户账号 CSV 新增字段：

- 运营风险标签
- 运营备注数
- 最近运营备注

导出仍走 `/admin/exports/users.csv`，并写入 `data.export.download` 审计。

## 8. 移动端影响

无直接移动端展示影响。

移动端不会通过 `/me`、`/app/config` 或其他 C 端接口收到后台备注和风险标签。

间接影响：

- 运营看到标签后可能进一步执行处罚、回访、工单回复或通知。
- 这些后续动作才会对移动端产生可见变化。

## 9. 用户业务数据清理关系

`POST /admin/users/{phone}/clear-business-data` 不清理：

- 后台运营备注。
- 后台风险标签。

原因：清理业务数据主要用于测试阶段重置宠物、AI、社交、地点、通知、工单等业务状态；运营备注和风险标签属于后台风控/客服上下文，应保留。

## 10. 待澄清

1. 生产期是否允许客服角色编辑风险标签，还是只允许运营管理员和社区安全员？
2. 是否需要给风险标签增加颜色等级，例如普通、警示、高危？
3. 是否需要系统自动标签，例如高频被举报、高频 AI 失败、短信高频？
4. 备注是否需要支持关联对象，例如工单 ID、举报 ID、AI 任务 ID？
5. 备注是否需要支持作废/撤销，而不是只追加？
