# 工单质量统计与批量复盘说明

日期：2026-07-01

## 目标

把工单中心从“能分配和回复”推进到“能复盘客服质量”。

本次覆盖：

- 客服质量统计。
- 按负责人拆分的基础绩效视角。
- 排班冲突检测。
- 批量处理复盘。
- 移动端评分、用户重开和后台审计日志进入同一套统计口径。

## 后台入口

后台：工单中心。

新增区块：

- 客服质量统计。
- 批量处理复盘。

接口：`GET /admin/tickets`。

新增返回：

- `quality`：整体质量统计。
- `quality.byAssignee`：按负责人拆分的统计。
- `rosterConflicts`：客服排班冲突。
- `batchReview`：最近批量处理复盘。

## 指标口径

- 首响率：已有真实客服回复的工单 / 全部工单。
- 首响达标率：首响在配置中心 `support.firstResponseSlaHours` 内完成的比例。
- 分配或接单不算首响，只有客服真实回复用户才算首响。
- 解决达标率：`resolved` / `closed` 工单在配置中心 `support.resolutionSlaHours` 内完成的比例。
- 满意度：移动端用户在工单结束后提交的 1-5 分评分。
- 重开率：用户在移动端重开工单的比例。
- 未首响开放：未关闭且尚无客服回复的工单。
- 按负责人统计：使用当前工单 `assignee` 和配置中心负责人枚举关联；历史未知负责人会保留为历史负责人。

## 排班冲突

排班来自配置中心 `support.assignees`。

检测规则：

- 只检测启用中的负责人。
- 同一天存在重叠值班窗口即视为冲突。
- 跨午夜班次会拆成两个日内片段再检测。
- 冲突只提示运营，不阻断保存配置；避免测试期因为临时重叠值班无法发布配置。

## 批量复盘

批量处理仍只开放：

- 批量分配。
- 批量转处理中。
- 批量等待用户。
- 批量标记已解决。
- 批量关闭。
- 批量改优先级。

批量回复用户仍不开放，避免误触达。

每次批量动作会额外写入一条 `ticket.batch.update` 审计，记录：

- 动作类型。
- 工单数。
- 成功数。
- 失败数。
- 操作原因。
- 管理员和时间。

单个工单仍保留自己的 `ticket.update` before/after 审计。

## 移动端联动

本次不新增移动端页面，但统计不是后台孤立数据：

- 移动端提交 `/feedback` 后自动生成工单。
- 移动端 `/support/tickets/{ticketId}/rate` 评分进入后台满意度。
- 移动端 `/support/tickets/{ticketId}/reopen` 重开进入后台重开率。
- 移动端补充回复会把工单状态转回 `reviewing`，影响未关闭和等待用户统计。

## 仍需业务确认

- 周/月 KPI 是否按自然周、自然月，还是按客服排班周期。
- 是否需要区分一线客服、安全客服、技术处理人。
- 低分评价是否自动生成质检任务。
- 外包客服结算是否使用后台工单数据。
- 是否开放批量回复审批；若开放，需要审批人、发送预览和撤回策略。

## 验证

```bash
node scripts/smoke-ticket-sla-roster.cjs
```

覆盖：

- 排班冲突返回。
- 分配后不算首响。
- 客服回复后进入首响统计。
- 用户评分进入满意度。
- 按负责人统计可回算。
- 批量处理写入复盘审计。
