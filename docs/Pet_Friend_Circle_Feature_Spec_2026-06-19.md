# Lumii 宠友圈功能文档 - 2026-06-19

> 2026-07-14 实现态补充：本文保留早期产品推导，冲突处以本补充、当前 RN 实现和 `API_Contract_MVP_v0.md` 为准。现版本已上线开发完成：发现页双 Tab、最多 6 图、本人归档、评论、点赞、招呼/约遛、举报/拉黑、腾讯云机审、后台人工复核、处罚/申诉和审核结果通知；已移除心情标签与 AI 润色。

## 1. 功能定位

宠友圈是 Lumii 的宠物生活动态流，形态接近微信朋友圈，但内容主体从“人发生活”收敛为“围绕宠物的小事”。

它不是泛社交广场，也不是纯附近列表，而是把用户已经在「今日小事」里记录的内容，进一步沉淀成可被附近宠友浏览、互动、破冰的宠物生活流。

一句话定义：

> 用户记录自家宠物的小事，附近宠友可以看到、点赞、评论、打招呼，并由此自然进入聊天或约遛。

## 2. 和现有功能的关系

当前已有基础：

- 「今日小事」发布页：已有 Figma Make 设计和 RN 页面，可继续复用为宠友圈发布入口。
- 「附近宠友小事」首页卡片：已有四态视觉，用来在宠物首页展示附近宠友动态摘要。
- 「发现页」：已有附近用户列表、定位、距离、打招呼、约遛。
- `POST /social/moments`：已支持发布小事到附近小事流。
- `GET /social/nearby-moments`：已支持读取附近其他用户的小事。

宠友圈要做的是把这些散点串成一个完整社交流：

```txt
记录今日小事 -> 同步到宠友圈 -> 附近宠友浏览 -> 点赞/评论/打招呼 -> 聊天/约遛
```

## 3. MVP 目标

MVP 阶段先做“可看、可发、可互动、可回流”的闭环。

### 3.1 用户价值

- 让用户有理由持续记录宠物日常，而不是只完成建档。
- 让附近社交不只依赖陌生人列表，而是通过宠物小事破冰。
- 让宠物首页的「附近宠友小事」不只是装饰，而能跳到完整宠友圈。

### 3.2 MVP 不做

- 不做复杂推荐算法。
- 不做关注关系流，先以附近流为主。
- 不做转发。
- 不做公开话题广场。
- 不做视频发布，先图片和文本。
- 不做复杂分组可见，先做基础可见范围。

## 4. 信息架构

### 4.1 一级入口建议

推荐把现有底部导航「发现」升级为社交聚合页，内部包含两个 Tab：

- 宠友圈：动态流，默认进入。
- 附近伙伴：原发现页附近宠物主人列表。

原因：

- 宠友圈和附近伙伴都属于“人的社交”，放在同一个一级入口更自然。
- 首页「附近宠友小事」点击「去宠友圈」可以进入发现页的宠友圈 Tab。
- 原发现页逻辑不丢，只是从一级页面变成发现页内的一个 Tab。

备选方案：

- 底部导航新增「宠友圈」，原「发现」保留。
- 不推荐 MVP 使用，因为会挤压底部导航，也会让发现/地图/消息/我的的层级变复杂。

### 4.2 页面结构

```txt
发现 / 宠友圈聚合页
  - Tab 1：宠友圈动态流
    - 发布入口
    - 附近小事列表
    - 点赞、评论、打招呼
    - 空态、加载态、失败态
  - Tab 2：附近伙伴
    - 沿用当前发现页
    - 筛选、搜索、打招呼、约遛

发布今日小事页
  - 沿用当前页面
  - 增加可见范围、是否同步宠物日历等轻量选项

动态详情 / 评论区
  - MVP 可先用 Bottom Sheet
  - 后续再独立页面
```

## 5. 核心用户流程

### 5.1 浏览宠友圈

1. 用户进入「发现」。
2. 默认展示「宠友圈」Tab。
3. App 自动获取当前位置。
4. 服务端按后台 `3km / 5km / 10km` 档位返回范围内、7 天内、开启附近可见的猫狗小事；首发默认 10km。
5. 用户可以下拉刷新。
6. 用户可以切换到「附近伙伴」查看完整宠物主人列表。

### 5.2 发布小事

1. 用户从首页「记录今日小事」、宠友圈顶部发布入口进入。
2. 页面沿用当前「今日小事」发布页。
3. 用户填写文本并添加 0-6 张照片；图片格按朋友圈方式逐格增加并显示 `x/6`。
4. 用户可选择可见范围：
   - 附近宠友可见，默认。
   - 仅自己可见。
5. 发布成功后：
   - 按用户选择写入宠物日历备忘。
   - 如果选择附近可见，同步到宠友圈。
   - 回到宠友圈或首页时可看到状态反馈。

### 5.3 点赞

1. 用户点击动态卡片上的点赞。
2. 前端即时高亮并更新数量。
3. 后端保存点赞状态。
4. 对方收到一条互动通知。
5. 再点一次取消点赞。

MVP 口径：

- 点赞可以重复切换。
- 点赞通知可做去重：同一用户对同一小事只保留最新状态。

### 5.4 评论

1. 用户点击评论。
2. 打开评论 Bottom Sheet。
3. 输入评论并发送。
4. 评论出现在该动态下。
5. 对方收到互动通知。

MVP 口径：

- 评论仅支持文本。
- 评论最多 140 字。
- 评论暂不支持楼中楼回复。
- 评论可删除自己的评论。

### 5.5 打招呼

1. 用户在某条小事上点击「打招呼」。
2. 复用当前打招呼逻辑。
3. 如果双方已有会话，则进入对话。
4. 如果是第一次互动，则进入招呼请求。

### 5.6 进入聊天

1. 宠友圈动态卡片展示宠物名、主人昵称、模糊距离。
2. 点击头像/昵称可打开轻量资料卡。
3. 资料卡内可打招呼、发消息或约遛。

MVP 可以先不做完整资料页，复用现有发现页主人卡逻辑。

## 6. 动态卡片内容

### 6.1 字段

```ts
type PetCirclePost = {
  id: string;
  ownerId: string;
  ownerName: string;
  ownerAvatarUrl?: string;
  petId: string;
  petName: string;
  petAvatarUrl?: string;
  species: 'cat' | 'dog';
  text: string;
  imageUrls: string[];
  distance: string;
  createdAt: string;
  likeCount: number;
  likedByMe: boolean;
  commentCount: number;
  visibility: 'nearby' | 'private';
};
```

### 6.2 卡片展示

推荐展示：

- 宠物头像。
- 宠物名 + 主人昵称。
- 模糊距离，例如 `500m 内`、`约 1-2km`。
- 时间，例如 `刚刚`、`12分钟前`、`昨天 18:30`。
- 文本，默认最多展示 4 行，可展开。
- 图片，最多 6 张；1、2、3、4-6 张分别使用稳定网格布局。
- 互动栏：点赞、评论、打招呼。

### 6.3 内容口径

动态的语气应偏生活、温暖、轻社交。

示例：

- `Lucky 今天第一次主动叼球过来，像是在说“快陪我玩！”`
- `奶油在楼下闻了十分钟树叶，像在认真读一份日报。`
- `团子今天晒太阳晒到打呼，猫猫的周五也太舒服了。`

## 7. 可见性与隐私

### 7.1 MVP 可见范围

发布时提供两个选项：

- 附近宠友可见：进入宠友圈，受附近范围限制。
- 仅自己可见：只进入宠物日历，不进入宠友圈。

### 7.2 附近规则

- 后台可选择 `3km / 5km / 10km`，首发默认 `10km`；服务端为权威范围，客户端参数不能越权修改。
- 附近伙伴按用户当前定位判断；附近小事按发布时的模糊位置快照判断，发布者移动后历史小事不跟随迁移。
- 使用模糊距离，不展示精确坐标。
- 只展示开启「附近可见」的用户。
- 用户关闭「附近可见」后：
  - 不再被别人看到。
  - 自己发布的附近可见小事从附近流隐藏。
  - 历史小事可保留在自己的宠物日历/个人记录中。

### 7.4 多宠身份与归档规则

- 小事、评论、点赞、举报、招呼和约遛在创建时固化动作宠物 ID；用户随后切换当前宠物，不重写历史身份。
- “我发布的小事”只展示当前宠物的归档；同账号其他宠物的小事不混入。
- 双方接受招呼后可查看对方当前宠物的宠友圈；他人页面不显示“我”、换封面或删除权限。
- 删除单只宠物会永久删除其宠物日历、AI、媒体和宠友圈从属数据；已接受的账号级社交关系保留，待处理的宠物级请求清理。

### 7.3 删除规则

- 用户可以删除自己发布的小事。
- 删除后：
  - 宠友圈不再展示。
  - 关联点赞、评论不再展示。
  - 是否保留宠物日历记录需要二选一：
    - 当前实现：删除宠友圈动态不删除宠物日历记录。
    - 后续可在删除弹窗里提供“同时删除健康记录”选项。

## 8. 状态与异常

### 8.1 宠友圈列表状态

至少需要以下状态：

- 加载中。
- 有动态。
- 附近有伙伴但暂无动态。
- 附近无伙伴/暂无动态。
- 定位未授权。
- 附近可见关闭。
- 网络失败。
- 下拉刷新中。

### 8.2 发布状态

沿用当前「今日小事」状态，并补充：

- 发布中。
- 发布成功。
- 发布失败，保留草稿。
- 文本为空。
- 图片上传失败。
- 附近可见关闭时，提示只能保存为仅自己可见。

### 8.3 互动状态

- 点赞中。
- 点赞失败，回滚状态。
- 评论发送中。
- 评论失败，保留输入内容。
- 小事已删除，提示刷新列表。
- 对方关闭附近可见，打招呼失败并提示刷新。

## 9. 后端接口草案

当前已有：

- `GET /social/nearby-moments`
- `POST /social/moments`

为了完整宠友圈，建议演进为：

### 9.1 获取宠友圈动态

```txt
GET /social/pet-circle/posts?lat=23.1291&lng=113.2644&radiusKm=10&cursor=xxx
```

返回：

```ts
{
  items: PetCirclePost[];
  nextCursor?: string;
}
```

### 9.2 发布动态

```txt
POST /social/pet-circle/posts
```

请求：

```json
{
  "content": "今天 Lucky 主动叼球来找我玩",
  "imageUrls": ["https://media.example/pet-circle-001.jpg"],
  "visibility": "nearby",
  "syncToHealthCalendar": true
}
```

### 9.3 点赞/取消点赞

```txt
POST /social/pet-circle/posts/{postId}/like
DELETE /social/pet-circle/posts/{postId}/like
```

### 9.4 评论

```txt
GET /social/pet-circle/posts/{postId}/comments
POST /social/pet-circle/posts/{postId}/comments
DELETE /social/pet-circle/comments/{commentId}
```

### 9.5 删除动态

```txt
DELETE /social/pet-circle/posts/{postId}
```

## 11. 与宠物日历的关系

「今日小事」可以同步为宠物日历备忘。宠友圈与宠物日历是两个概念：

- 宠物日历记录：给自己看，偏记录和回顾。
- 宠友圈动态：给附近宠友看，偏社交和互动。

发布页可以保留一个入口，但底层应拆两份数据：

```txt
发布今日小事
  -> syncToHealthCalendar=true：原子创建宠物日历备忘
  -> visibility=nearby：创建宠友圈动态
```

这样可以避免用户删除社交动态时误删自己的宠物日历记录。

## 12. 通知规则

宠友圈相关通知进入现有通知中心。

通知类型建议：

- `pet_circle_like`：有人赞了你的小事。
- `pet_circle_comment`：有人评论了你的小事。
- `pet_circle_greeting`：有人从你的小事发起打招呼。

MVP 通知展示：

- 标题：`Lucky 的小事有新互动`
- 内容：`安安赞了这条小事`
- 点击后：
  - 如果有动态详情页，进入详情。
  - 如果没有详情页，进入宠友圈并定位到该动态附近。
  - 当前 RN 实现：点赞/评论/从动态打招呼通知携带 `postId`（评论通知额外携带 `commentId`），点击后进入宠友圈、清除搜索筛选、将对应动态临时置顶并显示“来自通知”高亮。

## 13. 内容安全与社区规则

当前生产基座：

- 手机号、微信号、QQ、邮箱、外部链接。
- 辱骂、灰产、广告。
- 过长文本。
- 空内容。

- 腾讯云文本与图片内容安全，按小事、评论、头像/宠物图和地点内容使用独立 BizType。
- 评论举报。
- 用户拉黑与黑名单管理。
- 动态举报。
- 被举报内容对举报者即时隐藏。
- 风险内容进入后台审核队列；允许、待复核、驳回/隐藏均有移动端状态和结果通知。
- 后台支持证据快照、分配/SLA、批量处理、处罚建议、双人会签、举报申诉与处罚申诉。

正常低风险内容通过腾讯云后直接发布，不要求每条人工复核；只有机审 `review`、用户举报、风控升级或运营抽检进入人工队列。

## 16. MVP 验收标准

- 用户 A 发布一条附近可见小事。
- 用户 B 在后台配置半径内刷新宠友圈，可以看到用户 A 的小事；首发默认范围为 10km。
- 用户 A 发布后移动到另一座城市，原小事仍锚定原发布地点，不会出现在新城市附近流；但仍保留在 A 的个人宠友圈归档。
- 客户端传入大于后台配置的半径时，服务端仍按后台范围过滤。
- 用户 B 点赞后，用户 A 的通知中心出现互动通知。
- 用户 B 评论后，用户 A 可看到评论通知。
- 用户 B 从动态发起打招呼，沿用现有招呼请求/会话逻辑。
- 用户 A 关闭附近可见后，用户 B 刷新宠友圈不再看到 A 的小事。
- 用户 A 删除宠友圈动态后，B 刷新后不再看到；A 的宠物日历记录仍保留。
