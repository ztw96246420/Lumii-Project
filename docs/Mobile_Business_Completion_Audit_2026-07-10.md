# 移动端业务完整性与上线收口审计

日期：2026-07-10

## 1. 审计口径

本轮目标不是只确认页面能打开，而是同时检查：

- 移动端路由是否完整、是否存在不可直达或未渲染页面。
- 移动端操作是否调用真实对应 API，Web 预览是否与服务器数据严格隔离。
- 用户端动作是否能在运营后台形成可处理、可追踪、可审计的闭环。
- 关键正常态、空状态、异常态和恢复动作是否进入自动化回归。
- 正式上线仍依赖哪些生产配置、外部服务和业务签署。

## 2. 本轮已完成

### 2.1 移动端路由完整性

- `AppRoute` 当前共 39 个路由。
- Web 预览路由改为 `Record<AppRoute, true>` 穷举映射；以后新增路由但未加入预览映射时，TypeScript 会直接报错。
- 所有 39 个路由都能进入现有 `renderScreen` 分支。
- 新增直达验收覆盖：灵伴生成中、我发布的小事、备忘、我的反馈、无宠物、上传、识别结果、编辑主人资料、宠物档案、账号安全。

### 2.2 Web 预览与真实服务器隔离

- `ai`

- `auth`
- `avatar`
- `config`
- `health`
- `messages`
- `permissions`
- `pets`
- `places`
- `settings`

- `support`

本轮修复了反馈工单、处罚申诉、举报申诉、招呼请求举报、登录、验证码、权限保存、退出和配置读取的预览直连残留，避免设计预览或 Playwright 操作误写测试服务器。

### 2.3 客服工单前后台闭环

自动化已覆盖：

1. 用户提交反馈并生成客服工单。
2. 用户在客服首次回复前补充说明。
3. 后台读取到 `reviewing` 状态并分配客服。
4. 客服回复并解决工单，用户收到站内通知。
5. 用户评价处理结果。
6. 用户重开已解决工单。
7. 后台读取重开次数并再次解决。
8. 用户补充、重开、客服回复、分配和批量操作全部写入审计日志。

移动端 Playwright 同时覆盖：招呼请求举报 -> 我的 -> 设置与隐私 -> 我的反馈 -> 反馈详情 -> 提交补充。

### 2.4 缺失头像不再伪造宠物形象

- 移除首页、聊天、宠友圈、约遛、招呼请求、多宠管理、宠物档案、安全中心等业务页面对内置金毛图片的生产回退。
- 真实头像缺失或图片加载失败时统一展示中性爪印缺省态。
- 内置金毛资源只保留给明确的 Web 设计夹具和 AI 生成预览，不再作为真实用户兜底数据。
- 新增 `mockAvatar=missing` 专用验收场景，同时断言圆形头像和首页大形象均进入中性缺省态。
