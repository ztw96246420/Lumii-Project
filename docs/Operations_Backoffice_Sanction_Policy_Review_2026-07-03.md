# 运营后台处罚策略复盘
日期：2026-07-03

## 背景

Lumii 已经具备警告、禁言、冻结、封禁、处罚申诉、举报处理申诉和举报处罚建议，但此前“用户处罚”页主要是创建处罚和查看流水，缺少一个策略层视角。

本次补齐处罚策略复盘，帮助运营判断：

- 哪些处罚模板命中最多。
- 哪些模板申诉率或申诉通过率偏高。
- 哪些处罚来源撤销率偏高。
- 哪些用户多次违规或同时存在多条生效限制。
- 举报成立后的处罚建议是否被及时处理。

## 本次实现

新增后台接口：

- `GET /admin/sanction-policy-review`

用户处罚页新增“处罚策略复盘”区域：

- 指标卡：处罚总数、申诉推翻率、当前受限用户、举报建议应用率。
- 运营建议：未处理申诉、高申诉模板、待应用举报处罚建议、重复违规用户。
- 移动端影响说明：警告、禁言、冻结、封禁分别如何影响 App 写接口。
- 模板表现：按处罚模板统计命中、生效、撤销、申诉率、推翻率和策略状态。
- 来源质量：按 `source` 统计手动处罚、举报处罚、宠友圈作者处罚等来源的撤销/推翻表现。
- 重复违规用户：聚合多次处罚或多条生效限制的用户。
- 待处理举报处罚建议：展示举报处理后已生成但尚未应用的处罚建议。

## 数据口径

处罚复盘完全来自现有真实业务数据：

- `userSanctions`
- `sanctionAppeals`
- `socialReports[].sanctionSuggestion`
- 当前用户账号状态和生效处罚

不会新建独立策略状态，也不会为了复盘伪造样本。

## 移动端联动

该页面只读，但展示的移动端影响来自真实后端限制：

- `warning`：只通知和留痕，不限制移动端功能。
- `mute`：会拦截发布小事、评论、点赞、打招呼、约遛、私信、宠友圈封面、地点投稿和地点点评。
- `freeze` / `ban`：会拦截大多数写操作，但保留反馈、工单、通知已读、token refresh 和申诉入口。

因此复盘里的“当前受限用户”和生效类型，反映的是移动端下一次请求会遇到的真实限制。

## 当前不做

- 不自动升级处罚。
- 不自动批量处罚。
- 不替代申诉人工复核。
- 不做多管理员双人审批。

这些能力仍应在策略稳定后，配合审批流和更细角色权限再开放。

## 验证

新增烟测：

```bash
node scripts/smoke-sanction-policy-review.cjs
```

脚本覆盖：

- 举报成立生成处罚建议。
- 按建议创建处罚并限制移动端账号。
- 处罚策略复盘统计模板命中、来源质量和移动端限制。
- 用户提交账号限制申诉。
- 后台通过申诉并撤销处罚。
- 再次创建手动冻结，验证重复违规用户和推翻率统计。
